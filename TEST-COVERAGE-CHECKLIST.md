## 📝 测试编写规范

### 必须测试的场景

```javascript
// ✅ 好的测试结构
describe('组件名', () => {
  describe('空状态', () => {
    test('空输入时应该...', () => {})
    test('空数据时应该...', () => {})
  })
  
  describe('正常状态', () => {
    test('单条数据应该...', () => {})
    test('多条数据应该...', () => {})
  })
  
  describe('边界情况', () => {
    test('最小值应该...', () => {})
    test('最大值应该...', () => {})
    test('超长输入应该...', () => {})
  })
  
  describe('错误处理', () => {
    test('无效输入应该...', () => {})
    test('异常输入应该...', () => {})
  })
  
  describe('视觉/布局', () => {
    test('高度应该...', () => {})
    test('宽度应该...', () => {})
    test('颜色应该...', () => {})
  })
})
```

### 测试断言规范

```javascript
// ✅ 好的断言
expect(height).toBeGreaterThan(400)  // 明确的最小值
expect(width).toBeLessThan(2000)     // 明确的最大值
expect(color).not.toBe('transparent') // 排除错误值

// ❌ 坏的断言
expect(height).toBeTruthy()  // 太模糊
expect(width).toBeDefined()  // 没意义
```

## 🎯 本次教训

### 问题根源

1. **思维定势**: 认为"功能正常 = 一切正常"
2. **测试盲区**: 只测功能，不测视觉
3. **缺少规范**: 没有强制要求测试空状态
4. **过度自信**: 认为简单功能不需要全面测试

### 改进方向

1. **建立清单**: 每次开发对照检查
2. **强制要求**: 空状态测试必须覆盖
3. **视觉测试**: 添加视觉回归测试
4. **人工验证**: 自动化 + 手动双重验证
